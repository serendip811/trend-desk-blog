# Lessons

- 정적 페이지를 여러 HTML 파일에서 직접 중복 관리하면 홈, 아카이브, 카테고리, RSS, 사이트맵이 쉽게 어긋난다. 글 메타데이터는 반드시 단일 원본에서 관리한다.
- 목록 UI와 본문 UI의 요구사항을 분리한다. 글 본문에 복원한 이미지를 목록 썸네일까지 자동 노출하지 않는다.
- 공용 피드 레이아웃을 유지할 때는 홈과 아카이브, 카테고리의 마크업 구조를 맞춰야 한다. 같은 클래스 이름이라도 DOM 구조가 다르면 레이아웃이 깨진다.
- 중복 주제 글을 즉시 삭제하기보다 `listed`, `rss`, `sitemap` 같은 공개 여부 플래그로 먼저 통제하면 기존 링크를 덜 깨뜨리고 정리하기 쉽다.
- 공개 글에는 `로컬 파이프라인`, `에이전트`, `초안`처럼 내부 제작 과정을 드러내는 문구를 넣지 않는다. 독자 기준의 문제와 효용만 남긴다.
- 사용자가 공식 링크를 직접 주면 그 링크를 본문 근거와 하단 링크에 우선 반영한다. 2차 요약보다 공식 원문을 먼저 연결한다.
- 트렌드 글은 `pipeline.py` 결과로 주제만 찾고, 본문 사실은 최신 공식 링크와 재확인 가능한 공개 데이터로 다시 쓴다. 날짜, 가격, 신청 기간, 자격 조건처럼 바뀔 수 있는 값은 출처 날짜를 함께 적고 공식 페이지 재확인을 유도한다.
- 절차 설명, 신청 안내, 요금제 비교처럼 시각 정보가 도움이 되는 글은 본문용 이미지를 기본 검토한다. 가능하면 최신 공식 캡처를 쓰고, 없으면 직접 만든 설명 이미지를 넣는다.
- 이미지가 들어가더라도 목록은 텍스트 중심을 유지한다. 이미지는 피드 썸네일보다는 글 본문 설명용으로 쓴다.
- SEO는 생성기와 원본 메타를 같이 관리해야 한다. 글 원본은 `title`, `page_title`, `description`, `summary`, `updated_at`, 공식 링크, 내부 링크를 책임지고, canonical/OG/JSON-LD/sitemap/robots 같은 공통 SEO 출력은 생성기에서 일괄 관리한다.
